About

What is Code Coach?

Whether you need to master algorithms for school, interviews, or just for fun, it is a journey that requires substantial practice.

This project was inspired by the difficulty of finding good practice problems that suited users both in difficulty and topics. Code Coach predicts a user's growth trajectory based on past behavior, and recommends optimal practice problems to accelerate improvement.

What is competitive programming?

Users from all over the world convene online to solve algorithms problems during programming contests. Competitive programming platforms serve as online judges that determine the correctness of user's solutions. At the end of each contest, users are assigned ratings as a measurement of their skill with respect to every other user.

These platforms measure users' programming skills over time, and keep track of user actions such as submitting correct or incorrect solutions to problems. Code Coach is currently built on top of data from Codeforces, one of the oldest and largest competitive programming communities. (If you are a member of Codeforces, you may remember one of my earlier blog posts, written jointly with chenmark.)

Details



This demo version of code coach is powered by a gradient boosted regressor, trained on 20 million Codeforces user submission entries, 1 million user rating changes, and 4,000 problem ratings and tags from 2011 to 2017. The final dataset consisted of around 100 features corresponding to around 800,000 next user rating change. Training and test sets were split in a stratified fashion based on user rating and the total number of problem solved, to obtain similar distributions of user types.

Information was retrieved from the Codeforces API. Feature extraction and hyperparameter gridsearch with 3-fold cross-validation was performed in parallel on an 8-core instance of Google Compute Engine. Training was performed using XGBoost.

While this gradient boosted regressor yielded a r2 value of 0.41 on the test set, the best fit was provided by a random forest regressor, with a r2 value of 0.53 on the same test set. The random forest regressor was trained with Scikitlearn's gridsearchCV, yielding a best estimator of 100 trees. However, this model occupies 11 gigabytes and cannot be cost-effectively hosted on Amazon Web Services. Thus, the lighter (and faster) gradient-boosted regressor is used here instead.

Model details

Feature engineering

Features used in this model primarily capture how hard a user worked, and how much they challenged themselves.

The number of problems solved over the last month is a highly important feature in the final model.

Not all solves are equal. Typically, the larger the difference between a user's rating and a problem's rating, the more the user's score is projected to increase.

The user's current rating, as well as averaged rating over the last several months are also highly predictive of next rating changes. However, since the past is outside of the user's control, these features are used for regression but not for recommendation.

Recommendations for each user are made by iterating over features within the user's control (such as the number of problems solved and the types of problems), perturbing one feature at a time, and querying whether the model will predict a more favorable rating change. Behaviors that result in better rating changes are then recommended to the user.

Piecewise Linear regression

The project initially started with a simple linear model, which, after extensive feature engineering, could capture r2 of around 0.2 on smoothed user rating data. A piecewise linear regression based on 20 separate bins of users yielded r2 values from 0.17 to 0.45, with highs mostly in lower-rated users, and poor performance on medium- to highly-rated users.

Neural Networks

Due to the highly nonlinear mapping of features to user rating changes, both a feed-forward neural network and a recurrent neural network were considered. A 3-layer densely connected neural network with 50 neurons each in the hidden layers produced a r2 value of 0.2, on par with linear regression. A recurrent neural network with GRUs ("simplified" LSTM) was attractive due to its ability to handle time series of different lengths and the occurence of many event types. Unfortunately, this network could not be trained and tuned in the time frame of this project.
